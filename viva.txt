BEGINNER LEVEL (Q1-Q34)
============================================================

Q1. What is the main purpose of this project?
A1. The project implements a desktop Hangman game using C++ and SFML. It demonstrates event-driven programming, object-oriented design, rendering, audio feedback, and state management. The player guesses letters to reveal a hidden word while wrong guesses progressively draw a hangman figure. The game includes menu navigation, hints, score tracking, and replay behavior.

Q2. Which file contains the main game logic?
A2. The main logic is in Hangman-trial.cpp. It defines classes, asset loading, event processing, rendering, and game flow. All gameplay transitions and user interactions are controlled from this file.

Q3. Which libraries are used and why?
A3. iostream is used for debug output. string is used for word and text manipulation. SFML/Graphics.hpp handles window, shapes, text, and event systems. SFML/Audio.hpp handles sound buffers and playback. stdlib.h plus ctime/time.h support randomness and failure constants. cctype is used for isalpha and toupper input normalization.

Q4. Why is srand(time(0)) used?
A4. srand(time(0)) seeds the pseudo-random generator with current time. Without seeding, rand() would produce the same sequence each run. Seeding helps vary selected words across sessions, improving replay quality.

Q5. How is a random word chosen?
A5. The code computes a random index c using rand() % 5. That index is used for both words[c] and hints[c], preserving word-hint alignment. This ensures the selected hint matches the chosen answer.

Q6. What does guessedString represent?
A6. guessedString stores the player's current progress. Initially it contains underscores with length equal to the answer word length. As correct letters are guessed, matching positions are replaced with actual letters.

Q7. How are underscores initialized?
A7. setGuessedString() clears guessedString and appends one underscore per character in correctString. This uses a loop from index 0 to length-1. It guarantees one-to-one positional mapping between hidden and revealed characters.

Q8. What is the role of the Game class?
A8. Game is a base class holding tries and shared scoring counters. It also provides isLost() logic and static members for max tries, wins, and losses. Hangman inherits from it to reuse common state.

Q9. Why are wins and losses static?
A9. Static members are shared by all instances of the class. This design preserves a session-wide score instead of per-object isolated counters. It also reflects that wins/losses are global game statistics.

Q10. How does the program detect a loss?
A10. Loss is checked using isLost(), which returns true when tries > max_tries. Since max_tries is 6, losing happens after the 7th wrong guess. This aligns with drawing all hangman parts before final failure.

Q11. What is the role of the Hangman class?
A11. Hangman manages word-specific state and guess processing. It stores correctString, guessedString, and hint. It exposes guessWord(), isWon(), and setters/getters for gameplay updates.

Q12. How is a win detected?
A12. isWon() compares guessedString and correctString directly. If every position has been correctly revealed, strings are equal and win state is true. This is a clean and reliable completion condition.

Q13. What is the importance of converting input to uppercase?
A13. The answer words are uppercase, so user input is normalized with toupper. This avoids case mismatch errors and allows both lowercase and uppercase keyboard input from users. It improves usability without changing stored words.

Q14. How are non-letter keys filtered out?
A14. The code checks ASCII range and then applies isalpha. This ensures only alphabetic letters are processed as guesses. Numbers, symbols, and control characters are ignored.

Q15. How does duplicate guess prevention work?
A15. A boolean array choice[] tracks whether each alphabet letter has already been used. The index is computed as uppercaseLetter - 'A'. If choice[index] is true, the program skips guess processing for that letter.

Q16. Why is choice sized larger than 26?
A16. The current array is larger than needed, but the useful range is 0 to 25 for A-Z. A tighter size of 26 would be cleaner and semantically correct. The existing approach still works because indexed accesses remain in the alphabet range.

Q17. What happens when a correct letter is typed?
A17. guessWord() scans the answer and reveals all matching positions. The displayed guessed text is updated immediately. Correct sound is played, and if word completion occurs, win flow triggers with score update.

Q18. What happens when a wrong letter is typed?
A18. Wrong sound plays and tries increase. Depending on try count, the next hangman shape is moved into visible position. After final threshold, loss text is shown and losses counter increments.

Q19. Why are many objects moved to position (1000, 0)?
A19. This is a visibility trick to hide UI elements by moving them outside the window viewport. Instead of destroying and recreating objects, the code repositions them. It simplifies state changes between menu and gameplay screens.

Q20. What is LegacyRect used for?
A20. LegacyRect is a simple custom rectangle with a contains(x, y) hit-test function. It is used to detect clicks on text-based button regions (Play, Rules, Close). It replaces older rectangle API dependencies.

Q21. Why is event polling placed in a loop?
A21. In event-driven frameworks like SFML, pollEvent drains queued OS/window events. Processing all pending events each frame prevents lag and missed input. It supports close events, mouse clicks, and text input reliably.

Q22. What does the rendering loop do each frame?
A22. It clears the window, draws background and UI elements, then displays the frame. This is the standard draw cycle in immediate-mode rendering systems. Repeating this cycle keeps visuals responsive and up to date.

Q23. What are the main text objects in the game?
A23. Text objects represent title, menu buttons, guessed word, prompt, rules title/body, hint, and score. They are configured with different fonts, colors, sizes, and positions. The code updates their strings dynamically based on game state.

Q24. How is hint text displayed?
A24. A hint string is selected with the word and then split into two lines using substr. The displayed text object is updated during round start. This gives contextual help to the player.

Q25. What does setTries(0) on Play do?
A25. It resets wrong-attempt count for a new round. Without reset, old round state would leak and cause immediate losses or incorrect drawing. Reset is essential for reproducible replay behavior.

Q26. Why does the game use sf::SoundBuffer and sf::Sound separately?
A26. SoundBuffer owns loaded audio sample data in memory. Sound is a lightweight playback object referencing a buffer. This separation supports efficient reuse and avoids reloading files per playback.

Q27. Why check loadFromFile return values?
A27. Asset loading can fail due to missing files or wrong paths. Checking return values allows early failure and prevents undefined behavior from invalid resources. It improves robustness and debuggability.

Q28. Why is there a playing boolean?
A28. playing gates keyboard input so letters are accepted only during active gameplay. On menu, rules, win, or loss screens, input should not alter guessed word logic. This boolean is a simple state guard.

Q29. Why keep positions in a 2D array?
A29. positions stores reusable coordinates for dynamic element placement and restoration. It centralizes fixed coordinates and reduces hardcoded duplication in transitions. This supports consistent layout updates.

Q30. What is the benefit of using classes here?
A30. Classes encapsulate data and behavior, making logic modular and maintainable. Game handles shared attempt/score concepts while Hangman focuses on word mechanics. This separation improves readability and future extension.

Q31. What is the significance of max_tries = 6 with loss at tries > max_tries?
A31. It effectively allows 7 wrong attempts before loss check returns true. This matches drawing sequence where final body parts appear before defeat message. The design should be explicitly documented to avoid confusion.

Q32. Which part of code reveals letter positions in the word?
A32. Hangman::guessWord(letter) loops over correctString and compares each character with typed letter. On matches, it reconstructs guessedString around index i, replacing underscore at that exact position. This is the core positional recognition logic.

Q33. How does Play Again work?
A33. After win/loss, the Play button text changes to Play Again and is moved to end-screen location. Clicking it re-runs setup: new random word/hint, reset tries, clear choice array, and hide/show relevant UI. This avoids restarting the application.

Q34. What key programming concepts does this project demonstrate?
A34. It demonstrates classes and inheritance, static members, event-driven architecture, state machines, string manipulation, real-time rendering, audio integration, and input validation. It also shows practical UI control with object positioning and gameplay loop design.

ADVANCED LEVEL (Q35-Q67)
============================================================

Q35. Analyze the time complexity of guessWord().
A35. guessWord() loops through n characters of the answer word, so base scanning is O(n). However, each replacement uses substring construction, which can cost O(n), making worst-case around O(n^2) per guessed letter for many matches. Given short words, this is acceptable, but for large strings a mutable index assignment approach would be more efficient.

Q36. How can guessWord() be optimized?
A36. Instead of rebuilding strings with substr, directly assign guessedString[i] = correctString[i] when matched. This reduces update cost to O(1) per match and total O(n) per guess. It also improves readability and reduces temporary allocations.

Q37. Why is static_cast<unsigned char> used before cctype functions?
A37. cctype functions like isalpha/toupper are undefined for negative signed char values (except EOF). Casting to unsigned char ensures values are in valid domain. This is a correctness and portability best practice in C++.

Q38. Is there any off-by-one risk in isLost()?
A38. isLost() uses tries > max_tries, so with max_tries=6, loss occurs at tries=7. If design intent is exactly six mistakes allowed, this is correct only if six represents last visible step before loss check. If intent is lose on sixth mistake, condition should be tries >= max_tries.

Q39. Why choose text-entered events over key-pressed events for letters?
A39. TextEntered represents character input semantics, handling keyboard layout and text-level input better than raw key codes. For language-agnostic letter processing this can be more user-friendly. Here it is additionally filtered to ASCII and alphabetic characters.

Q40. Explain state management strategy in this game.
A40. The code uses implicit state through booleans and object positions. playing controls active guessing; menu/rules/game-over are represented by moving text/shapes in or out of view. This works but can be improved with explicit enum-based finite state machine.

Q41. What are trade-offs of using global-like static score counters?
A41. Static counters are simple and shared, ideal for single-session total score. But they reduce flexibility for multiplayer or profile-based scoring, and can complicate testing with persistent state across test cases. Dependency injection or context objects would scale better.

Q42. Why might hardcoded paths be problematic?
A42. Relative hardcoded paths assume executable working directory and exact folder names. If run from different directory, loads may fail. Better approaches include configurable asset root, packaging rules, or runtime path discovery.

Q43. What is the role of RenderWindow in SFML architecture?
A43. RenderWindow combines window creation with rendering target capabilities. It integrates event queue handling and graphics display pipeline. In this game it is central for pollEvent, draw calls, and frame presentation.

Q44. Why are object creations outside the main loop?
A44. Creating heavy resources every frame is expensive and causes stutter. Fonts, sounds, textures, and static shapes are initialized once, then reused. Only dynamic properties (text string, position, visibility) are updated during loop.

Q45. How does the game avoid repeated body part drawing logic from scratch?
A45. It pre-creates all hangman shapes and only changes their positions to show/hide based on tries. This avoids allocations during gameplay and keeps rendering deterministic. It is a lightweight scene-state toggle mechanism.

Q46. Discuss potential bug risk in fixed hint substring splitting.
A46. Using substr(0,58) and substr(58,58) assumes hint length at least 58 for second part relevance. Short hints can produce empty second line or awkward formatting. A robust text wrapping function based on width would be superior.

Q47. Why is using namespace std generally discouraged in headers?
A47. It pollutes global namespace and can cause symbol conflicts. In source files it is less dangerous but still reduces clarity at scale. Explicit std:: qualifiers improve readability in large or shared codebases.

Q48. Compare composition vs inheritance for this design.
A48. Inheritance is used because Hangman is modeled as a specialized Game with tries and score behavior. Composition could also work by storing a GameState object inside Hangman. Composition is often preferred for flexibility, but this inheritance is simple and understandable for this scope.

Q49. How does event ordering affect behavior?
A49. Events are processed in the order they are polled. Mouse and text events in a frame can both occur, and logic depends on current state variables. Correct gating with playing avoids accidental text processing while in menu modes.

Q50. What is the effect of not checking font load success?
A50. If font loading fails, text may render incorrectly or not at all, reducing usability. Because no fallback or error handling exists, diagnosing issue becomes harder for users. Add explicit return checks and error logs as done for critical audio files.

Q51. Why use bool array for guessed letters instead of set<char>?
A51. bool array indexed by A-Z offers O(1) access and minimal overhead. It is faster and straightforward for fixed alphabet. set<char> is more general but introduces tree/hash overhead and unnecessary complexity here.

Q52. Is the game deterministic for testing?
A52. Not by default due to time-based random seeding. For deterministic tests, seed with fixed value (for example srand(1)) or inject RNG abstraction. This enables repeatable gameplay paths for verification.

Q53. Explain memory ownership of SFML objects used.
A53. Most resources are stack-allocated local variables in main scope and live until program exit. Sound objects reference buffers that outlive them, which is correct. No manual delete is used, reducing leak risk.

Q54. Can this code be unit-tested easily?
A54. Core logic in Hangman class is testable (guessWord, isWon, setGuessedString). UI/event code in main is harder to test due to side effects and framework coupling. Extracting gameplay controller logic into separate class would improve testability.

Q55. Why is choice array reset on new game?
A55. It clears historical guessed-letter memory from previous round. Without reset, valid new-round guesses could be incorrectly treated as duplicates. Reset ensures independent rounds.

Q56. What software design principle is visible in Hangman class?
A56. Encapsulation: word state and related operations are bundled in one class. Consumers use methods instead of directly mutating internal strings. This centralizes invariants and reduces accidental misuse.

Q57. Discuss rendering order significance.
A57. Draw order determines visual layering in SFML. Background is drawn first, then shapes/text on top. If order changes, important UI elements may become hidden behind others.

Q58. What is the difference between sound.play() and soundintro.pause() usage here?
A58. One-shot effects (correct/wrong/exit) are triggered with play(). Intro audio is paused when gameplay starts and resumed on round completion. This creates contextual audio feedback tied to state transitions.

Q59. Could the code support lowercase answer words?
A59. Yes, if either answers are normalized to uppercase before comparison or input is compared case-insensitively. Current logic assumes uppercase answer strings and uppercases input, so lowercase words would fail unless transformed.

Q60. Why are there multiple text style assignments one after another?
A60. Some objects call setStyle twice, so later call overwrites earlier style flags if not combined. To combine styles, bitwise OR should be used (for example Bold | Underlined depending on SFML version API). Current code may unintentionally keep only last style.

Q61. Explain robustness of mouse hit testing with LegacyRect.
A61. It is simple and fast for axis-aligned rectangular areas. However, if text is resized/repositioned dynamically, rectangle bounds must be manually synchronized. Using SFML text global bounds could reduce mismatch risk.

Q62. What are potential UX improvements for guessed letters?
A62. Show alphabet panel with color states (unused/correct/wrong), display remaining tries, and disable sound overlap with short debounce. Also show previously guessed letters explicitly to reduce cognitive load.

Q63. How does the code implement replay without restart?
A63. It reinitializes mutable state: answer, hint, guessed string, tries, guessed-letter flags, and visual placements. It does not destroy window or recreate heavy resources. This offers fast looped play sessions.

Q64. What concurrency concerns exist here?
A64. None significant in this single-threaded design. Audio playback happens via SFML internals, but game logic itself is sequential in one thread. This simplifies synchronization and state correctness.

Q65. What improvements would you propose for scalability?
A65. Load words from external file, separate model-view-controller layers, implement explicit GameState enum, add text wrapping utility, and replace repeated position code with helper functions. Also add logging, error dialogs, and configuration support.

Q66. What are the implications of using int for string length loops?
A66. string::length() returns size_t (unsigned). Using int may cause warnings and potential issues for very large strings. Prefer size_t for correctness and type consistency.

Q67. Why does event loop handle window close explicitly?
A67. GUI frameworks rely on event messages for user-initiated close actions. Handling close ensures graceful termination and resource cleanup. Ignoring it would make the application unresponsive to window manager requests.

ELITE LEVEL (Q68-Q100)

Q68. Evaluate this project using SOLID principles.
A68. Single Responsibility is partially followed in Hangman class, but main() violates it by mixing setup, input, state transitions, rendering, and audio policy. Open/Closed is weak because adding states requires editing central conditional blocks. Liskov is fine for simple inheritance. Interface Segregation and Dependency Inversion are limited because concrete SFML dependencies are directly embedded. Refactoring into separate controller/state/view systems would better align with SOLID.

Q69. Propose a finite state machine model for this game.
A69. Define enum class GameState { Menu, Rules, Playing, Won, Lost }. Transition on events: Menu->Playing by Play click, Menu->Rules by Rules click, Rules->Menu by Back/Play, Playing->Won/Lost based on checks, Won/Lost->Playing by Play Again. Rendering and input handling switch on state, eliminating hidden-via-position hacks and reducing accidental cross-state behavior.

Q70. How would you redesign input architecture for extensibility?
A70. Introduce InputHandler that translates SFML events into domain commands like StartGame, OpenRules, GuessLetter(char), ExitGame. GameController consumes commands and mutates model state. This decouples framework events from business logic and enables unit testing without GUI dependencies.

Q71. Explain why deterministic simulation matters in game testing.
A71. Deterministic behavior allows reproducing bugs exactly. If RNG and event timing are controlled, state transitions can be validated against expected snapshots. For this project, deterministic seed and scripted input sequence would verify win/loss logic and draw progression automatically.

Q72. What are the hidden costs of repeated substr concatenation?
A72. Each concatenation can allocate temporary strings, copy character ranges, and increase heap churn. In tight loops this degrades performance and cache locality. Direct index assignment in mutable guessedString avoids most overhead.

Q73. If you had to support Unicode letters, what changes are needed?
A73. Current ASCII and cctype logic is insufficient for Unicode alphabets. Use Unicode-aware text normalization and classification (for example ICU or locale-aware APIs), and map guessed-letter tracking to code points rather than A-Z index array. UI and fonts must also support glyph coverage.

Q74. Discuss exception safety and error handling quality.
A74. Current code uses return codes on some resource failures, which is basic but incomplete. No centralized error reporting exists, and partial loads can leave app in degraded state. A structured initialization phase with explicit failure messages and fallback policy would improve resilience.

Q75. How would you architect a data-driven content system?
A75. Move words and hints to external JSON/CSV/TXT and load at startup. Validate schema, remove duplicates, and support categories/difficulty tags. This allows non-code content updates and scaling to larger datasets.

Q76. Why is direct UI coordinate management brittle?
A76. Hardcoded coordinates are hard to maintain across resolutions and DPI settings. Small layout changes require many manual edits and risk inconsistencies. A layout manager or anchor-based positioning system improves adaptability.

Q77. Compare immediate mode rendering vs retained UI frameworks for this case.
A77. SFML rendering here is immediate mode: redraw everything each frame. It is simple and predictable for small games. Retained frameworks can reduce manual draw orchestration but often add complexity and less control for custom game visuals.

Q78. How would you implement undo for last guess?
A78. Store immutable snapshots or command history with inverse operations (letter guessed, previous tries, previous guessedString, previous visual state). Undo pops last action and restores state. Need careful handling to avoid undo across state boundaries like new game transitions.

Q79. Analyze current coupling between model and view.
A79. Coupling is high because main directly mutates both domain state and SFML objects in same blocks. This makes feature evolution riskier and debugging more complex. A presenter/view-model layer could map model state to draw properties cleanly.

Q80. How can you formalize correctness of win/loss logic?
A80. Define invariants: guessedString length equals correctString length; each revealed char must match correctString at same index; tries monotonically increases only on wrong new guesses; terminal states are mutually exclusive. Then verify transitions preserve invariants with tests and assertions.

Q81. What is your strategy to avoid magic numbers in this code?
A81. Replace literals like 900, 58, 1000, and coordinate constants with named constants or configuration structures. This improves readability and allows centralized tuning. It also clarifies semantic intent of thresholds and layout values.

Q82. How would you measure and optimize frame-time stability?
A82. Profile per-frame update and draw times using high-resolution timers. Avoid dynamic allocations during frame processing, cache text metrics, and minimize expensive string rebuilds. Maintain consistent loop workload and optionally cap frame rate.

Q83. Explain potential audio race/overlap concerns.
A83. Rapid repeated key presses can trigger overlapping sound plays, producing noisy feedback. Rate limiting or separate channels with stop/restart policy can improve clarity. Intro music pause/play transitions should ensure expected state when replaying rounds.

Q84. How would you implement adaptive difficulty?
A84. Add metadata per word: length, rarity, category complexity. Track player performance and select harder words for high accuracy or easier words after multiple losses. Dynamic hint richness can also be tuned by difficulty level.

Q85. What security concerns are relevant for this offline game?
A85. Direct attack surface is low, but malformed external files (if added later) should be validated. Avoid unchecked assumptions on data lengths and indexing. Defensive programming still matters for stability and reliability.

Q86. How can this game be made platform portable?
A86. Use CMake, avoid platform-specific APIs, keep resource paths relative, and package SFML dependencies per target OS. Continuous integration builds on Windows/Linux/macOS can validate portability early.

Q87. What does high cohesion mean in context of Hangman class?
A87. High cohesion means methods in Hangman all serve one concept: word-guess state transitions. This is mostly true for set/get, guessWord, and isWon. Keeping unrelated UI operations outside this class preserves cohesion.

Q88. How would you justify inheritance here to a strict reviewer?
A88. Shared properties tries/wins/losses and loss-rule behavior logically belong to a reusable game base abstraction. Hangman specializes by adding word and hint mechanics. For this educational project, inheritance improves conceptual clarity; for larger systems composition might be preferred.

Q89. Can this project benefit from constexpr or modern C++ features?
A89. Yes. Constants like max word count and coordinates can be constexpr. Use std::array for fixed arrays and enum class for states. Range-based loops and stronger typing improve safety and readability.

Q90. How would you implement localization (multi-language support)?
A90. Externalize all UI strings and hints into language resource files. Use locale selection at startup and Unicode-capable text handling. Keep domain logic language-agnostic and map translated strings in view layer.

Q91. Explain test cases for guessWord() that impress a professor.
A91. Include: single match, multiple matches, no match, repeated same guess, full-word completion edge, empty word safety, and mixed-case input normalization path. Also verify invariants: unchanged non-matching positions and no length mutation. Use deterministic assertions on guessedString after each step.

Q92. How would you refactor rendering code to reduce repetition?
A92. Store drawable pointers/references in containers per state and iterate to draw. Build helper functions like drawMenu(), drawGame(), drawRules(). This removes long repetitive draw call sequences and simplifies maintenance.

Q93. Discuss data validation for external word files if introduced.
A93. Validate non-empty words, alphabetic characters, max lengths, unique entries, and matching hint availability. Reject malformed records with logging. This prevents runtime errors and preserves gameplay quality.

Q94. How would you support variable window sizes responsively?
A94. Use relative positioning, anchors, and view transformations instead of fixed pixel literals. Recompute layout on resize events. Keep hitboxes tied to text bounds to maintain clickable correctness.

Q95. What are professional code-quality improvements you recommend first?
A95. Introduce explicit game states, replace magic numbers with constants, optimize guessWord indexing, add robust asset error handling, and split main into setup/update/render functions. Then add automated tests for Hangman logic.

Q96. How would you explain this project's architectural weaknesses honestly in a viva?
A96. It is functional and educational but has monolithic main() and implicit state transitions through position toggles. These choices accelerate development but reduce scalability. I would present a clear migration plan toward modular architecture and testability.

Q97. What is your complexity analysis of one full round?
A97. Let m be number of guesses and n word length. Input handling is O(m), guess checking roughly O(m*n) with current substring overhead potentially higher. Rendering is constant with fixed number of drawables each frame. For current tiny word set, performance is comfortably real-time.

Q98. If a professor asks "What did you do beyond basic implementation?" how do you answer?
A98. Highlight robust input filtering, duplicate guess prevention, synchronized word-hint selection, multi-screen flow, incremental visual feedback tied to tries, and audio-state integration. Mention object-oriented separation via Game/Hangman and reusable coordinate strategy. Then discuss planned refinements showing engineering maturity.

Q99. How would you defend design decisions under strict criticism?
A99. Emphasize fit-for-purpose choices for an educational desktop game: simplicity, readability, deterministic behavior, and immediate feedback. Acknowledge trade-offs and provide concrete upgrade path (FSM, MVC, tests, data-driven content). Demonstrating awareness of limitations and actionable improvements is often valued more than overengineering.

Q100. Give an elite-level final summary of technical learning from this project.
A100. This project teaches how domain logic, UI rendering, and event handling interact in real-time software. It connects C++ fundamentals (classes, inheritance, strings, arrays, static members) with systems concerns (asset lifecycle, input sanitization, state control, and feedback loops). Most importantly, it reveals the gap between "working code" and "scalable architecture," preparing the developer for professional refactoring discipline.
